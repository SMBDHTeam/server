# API 개발 규칙

## 기준

`docs/API_SPEC.md`가 현재 API 계약이며 `docs/API_FIELD_GUIDE.md`가 각 필드의 자료형과 의미를 정의한다.

구현은 다음 항목을 명세와 일치시켜야 한다.

- URI와 HTTP Method
- Path와 Query Parameter
- 요청 JSON과 검증 조건
- 응답 JSON과 필드 자료형
- 성공 상태 코드와 오류 코드
- 인증 여부

## 계약 변경

API 계약을 변경할 때는 같은 작업에서 다음을 수행한다.

1. `docs/API_SPEC.md` 수정
2. `docs/API_FIELD_GUIDE.md` 수정
3. `docs/api-change-log.md` 기록
4. 관련 테스트 수정
5. PR에 호환성 영향 표시

필드 삭제·이름 변경·자료형 변경, 필수 여부 강화, URI·Method·상태 코드·인증 변경은 호환성 파괴 변경이다. 응답 필드 추가도 클라이언트의 엄격한 역직렬화 가능성을 고려해 계약 변경으로 기록한다.

## 요청과 응답

- API 경계에는 DTO를 사용하고 Entity를 직접 반환하지 않는다.
- 날짜는 `YYYY-MM-DD`, 시간은 `HH:mm`, 날짜시간은 ISO-8601을 사용한다.
- 좌표는 WGS84 기준이며 `longitude`가 경도, `latitude`가 위도다.
- 배열은 값이 없을 때 가능한 한 `null` 대신 빈 배열을 반환한다.
- Enum 값은 명세에 허용값을 기록한다.
- 1차 스프린트의 일정 목록은 의도적으로 페이지네이션을 사용하지 않는다.
- 향후 일정 수 증가 또는 사용자 도메인 도입 시 별도 계약 변경으로 페이지네이션을 검토한다.

## 1차 스프린트 인증

- 모든 명세 API는 인증 없이 호출한다.
- 일정 ID를 아는 사용자는 일정을 조회하고 수정할 수 있다.
- 공유 링크 생성과 폐기도 인증 없이 처리한다.
- Spring Security를 사용하는 동안 명세 API가 실제로 익명 호출되도록 Security 설정에서 명시적으로 `permitAll` 처리한다.
- CSRF 정책은 JSON API 호출 방식과 테스트 환경을 고려해 명시적으로 구성하고, 기본 설정에 맡기지 않는다.
- JWT와 사용자별 권한은 사용자 도메인 도입 시 별도 변경으로 추가한다.

## 오류 응답

```json
{
  "code": "SCHEDULE_NOT_FOUND",
  "message": "일정을 찾을 수 없습니다.",
  "fieldErrors": [],
  "traceId": "01J..."
}
```

- `code`는 클라이언트가 분기할 수 있는 고정 문자열이다.
- `message`는 사용자에게 노출 가능한 문장으로 작성한다.
- `fieldErrors`는 입력 필드 오류 목록이다.
- `traceId`는 서버 로그 추적용 ID이며 비밀값을 포함하지 않는다.
- 내부 예외명과 스택 트레이스를 응답하지 않는다.

## 외부 API

- 외부 요청·응답 DTO와 내부 API DTO를 분리한다.
- 타임아웃, 응답 누락, 호출 제한, 인증 실패를 내부 오류 코드로 변환한다.
- 외부 API 원본 값이 불완전할 수 있는 필드는 nullable 또는 원문 보관 정책을 따른다.
- API Key는 환경 변수로 주입하고 요청 URL 전체를 로그에 남기지 않는다.
- 외부 Provider의 필드 변경이 내부 API 계약을 자동으로 바꾸게 하지 않는다.
