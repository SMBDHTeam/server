# AI 기반 일정 Planner 전환 계획

## 현재 상태

현재 일정 생성기는 AI가 일차별 장소 구성을 제안하고 결정론적 Planner가 현실 경로와 필수 제약을 검증하는 하이브리드 구조다.

- 기본 설정에서는 `RuleBasedPlanningPromptInterpreter`가 자유 요청에서 정해진 한국어 키워드만 찾는다.
- `AI_PLANNER_ENABLED=true`이고 API 키가 있으면 OpenAI Structured Outputs로 자유 요청을 해석하고, 제한된 후보 ID에서 일차별 장소 구성을 제안한다.
- AI 제안은 날짜 수, 방문 수, 중복, 필수 방문지, 고정 행사 날짜, 식사 슬롯을 서버에서 재검증한다.
- `PlacePreferenceScorer`가 질문 답변, 장소명, 거리 기반 고정 점수를 계산한다.
- `PlaceCandidateProvider`가 목표 방문지보다 최대 6곳 큰 후보 풀을 구성한다.
- `MultiDayPlanOptimizer`가 AI 제안과 함께 비교할 규칙 기반 Top-K 장소·날짜 배치안을 생성한다.
- `DayRouteOptimizer`가 일반 활동과 식사를 합쳐 하루 최대 5곳의 방문 순열을 비교한다.
- ODsay와 TMAP은 이동 경로 데이터 Provider이며 AI 모델이 아니다.

따라서 AI는 실제 일정의 장소·날짜 구성을 생성한다. 다만 방문 순서, 실제 대중교통 경로, feasibility와 Hard Gate는 재현 가능하고 검증 가능한 결정론적 Planner가 최종 확정한다.

## 목표 구조

AI와 최적화기의 책임을 분리한 하이브리드 구조를 사용한다.

```text
사용자 입력
→ AI Preference Interpreter
→ 구조화된 소프트 선호와 설명
→ Candidate Pool Builder
→ AI 일차별 장소 제안 + 규칙 기반 Top-K 탐색
→ 실제 경로 계산
→ Hard Gate
→ 품질 점수 비교
→ 최고 점수 일정 저장
```

### AI가 담당하는 것

- `customPrompt`를 허용된 선호 코드와 가중치로 변환
- 허용된 후보에서 질문 답변·자유 요청에 맞는 일차별 장소 조합 제안
- 필수 장소와 식사 슬롯을 포함한 날짜 배치 제안
- 일정 제안 신뢰도와 짧은 선택 요약 반환
- 인식하지 못한 요청과 낮은 신뢰도 반환

### 결정론적 Planner가 담당하는 것

- 필수 방문지, 고정 행사, 종료 제약 보존
- 운영시간, 이동시간, 일차 가용시간 검사
- AI 출력 계약 검증과 규칙 기반 대체 장소 조합 탐색
- 날짜별 방문 순서 탐색
- ODsay·TMAP 실제 경로 비용 계산
- Hard Gate 통과 일정만 비교·저장
- 같은 입력과 같은 AI 결과에 대한 재현 가능한 선택

AI 출력은 Hard Constraint를 추가하거나 기존 제약을 제거할 수 없다. AI 호출 실패, 시간 초과 또는 출력 검증 실패 때는 규칙 기반 Top-K로 생성이 계속된다.

## 최적화 목표

Planner 목적함수는 다음 우선순위를 갖는다.

1. Hard Gate 위반 수를 0으로 만든다.
2. 필수 방문지와 고정 행사를 모두 포함한다.
3. 실제 이동·대기·체류시간 안에서 품질 점수를 최대화한다.
4. 긴 이동, 도보, 환승, 미사용 시간을 줄인다.
5. 질문 답변과 AI가 해석한 소프트 선호를 최대한 반영한다.
6. 점수가 같으면 외부 API 호출과 생성시간이 작은 결과를 선택한다.

현재는 최대 20개 후보에서 장소 선택과 날짜 배치를 함께 탐색하고, 상위 다일 배치안과 일차별 상위 방문 순서를 실제 경로로 다시 평가한다. 외부 호출 예산 때문에 탐색 공간 전체를 실제 경로로 평가하지 않으므로 수학적 전역 최적해를 보장하지는 않는다.

## 단계별 구현

### 1. 해석 경계 분리

- 상태: 구현 완료
- `PlanningPromptInterpreter` 인터페이스 사용
- 현재 `RuleBasedPlanningPromptInterpreter`를 기본 구현으로 유지
- 해석 결과를 허용된 코드, 가중치, 신뢰도, 근거로 제한

### 2. AI 해석기 도입

- 상태: 1차 구현 완료. 실제 키를 사용하는 Provider 연동 검증은 별도 수행
- Provider와 모델을 설정으로 주입
- 구조화된 JSON Schema 출력만 허용
- 타임아웃, 재시도, 토큰·비용 상한 설정
- 원문 프롬프트를 애플리케이션 로그에 기록하지 않음
- 규칙 기반 fallback과 AI 사용 여부를 Preview의 `interpretedPrompt.source`로 기록

### 3. AI 일정 제안

- 상태: 구현 완료. 실제 OpenAI 키를 사용하는 운영 Provider 검증은 별도 수행
- 후보 ID, 일차 조건, 질문 답변, 필수 장소, 고정 행사, 식사 슬롯을 strict JSON Schema 입력으로 제한
- AI 응답의 후보 소속, 일차별 개수, 중복, 필수·고정·식사 조건을 서버에서 재검증
- AI 호출과 규칙 기반 후보 탐색을 병렬 실행하고 8초 read timeout 적용
- AI 제안이 실제 경로 비교에서 선택되면 `AI_GENERATED`, 규칙 후보가 선택되면 `AI_ASSISTED`로 기록
- 호출·출력 실패와 활성화 상태의 키 미설정은 `AI_FALLBACK`, 명시적 비활성화는 `RULE_BASED`로 기록

### 4. 후보 풀 탐색

- 상태: 2차 구현 완료. 후보 풀 최대 20개에서 정확 DP와 beam DP를 일정 크기에 따라 사용
- 목표 방문지 수보다 큰 상위 후보 풀 구성
- 필수 장소와 식사 후보는 별도 슬롯으로 보존
- 장소명·카테고리·콘텐츠 타입에서 체험 유형과 의미 그룹을 파생해 서로 다른 경험 후보를 우선 보존
- 장소 정체성·환경·가능 경험·콘텐츠·분위기의 다축 프로필을 계산해, 서로 다른 콘텐츠라도 실제 경험이 겹치는 후보를 감점
- 12곳 이하는 정확 DP, 초과 일정은 후보 다양성을 보존하는 beam DP로 장소 선택과 날짜 배치를 함께 탐색
- 외부 호출 없는 좌표 기반 경량 추정으로 상위 일정만 유지
- 식사 장소는 하루 3곳의 일반 활동 한도와 분리해 활성 시간창마다 최대 1곳을 추가한다.
- 테마 일치 보상은 일차 조합 안에서 `100% → 50% → 15% → 0%`로 포화시키고, 동일 체험의 일차·다일 반복은 별도 비용으로 감점한다.

### 5. 실제 경로 재평가

- 상태: 다일 날짜 배치안과 일차 방문 순서의 제한 폭 재평가 구현
- 동일 장소 집합에서 날짜 배정이 다른 DP 상태를 최대 3개 보존
- 다일 배치안은 일차 출발·도착 경계의 실제 대중교통 접근성과 일차 내부 좌표 추정 비용으로 비교
- 좌표 추정 상위 3개 순서와 단일 식사 슬롯의 위치별 대안을 ODsay 경량 경로로 비교
- 요청당 경량 Provider 호출을 기본 30회로 제한하고 날짜별로 균등 배분
- 선택된 구간은 ODsay 검색 결과를 재사용해 `loadLane`·실시간·도보 상세만 보강
- 일차 순서 비교에는 권역 전환·권역 재진입·1km 이상 구간의 방향 전환 비용을 포함한다. 좌표 기준 detour ratio는 기록만 하고 실제 Provider 기준선 확보 후 감점한다.
- 남은 작업: 선택되지 않은 후보와 점수 차이를 평가 로그에 기록

### 6. 검증

- 고정 시나리오에 AI 응답 Fixture를 사용해 재현성 확보
- AI 오류, 빈 응답, 허용되지 않은 코드, 시간 초과 fallback 테스트
- 동일 입력의 품질 평균과 최저점, p95 생성시간, 외부 호출 수 비교
- 실제 사용자 선택·수정 데이터를 이용한 오프라인 랭킹 평가 추가

## 완료 기준

- Preview에서는 `RULE_BASED`, `HYBRID_AI`, `FALLBACK` 해석 모드를, 일정 생성 운영 지표에서는 `AI_GENERATED`, `AI_ASSISTED`, `AI_FALLBACK`, `RULE_BASED` 생성 모드를 구분한다.
- AI가 없는 상태와 장애 상태에서도 모든 Hard Gate가 유지된다.
- 대표 시나리오 평균 85점 이상, 최저 80점 이상이다.
- 동일 후보 풀에서 기존 Planner보다 낮은 점수의 일정을 선택하지 않는다.
- 실제 Provider 반복 실행의 p95 60초 이하와 호출 상한을 성능 기준선에 기록한다.
