# 일정 Planner 성능 기준선

## 목적

일정 생성 리팩터링을 휴리스틱 개수로 평가하지 않고, 필수 제약 통과 여부와 사용자 품질 점수, 운영 비용으로 반복 비교한다.

## 판정 순서

1. Hard Gate를 검사한다.
2. Hard Gate를 모두 통과한 일정만 품질 점수를 계산한다.
3. 목표 방문지보다 최대 6곳 큰 후보 풀에서 장소 선택과 날짜 배치의 총비용을 최소화한다.
4. 같은 일차 장소 조합에서는 좌표 추정 상위 순서를 제한된 실제 경로 비용으로 재평가한다.
5. 품질 점수가 같으면 외부 API 호출 수와 응답시간이 낮은 구현을 우선한다.

## Hard Gate

| ID | 통과 기준 |
| --- | --- |
| `MUST_VISIT_PLACE_MISSING` | 모든 필수 방문 장소가 일정에 포함됨 |
| `EMPTY_DAY` | 여행 기간의 모든 일차에 방문 장소가 있음 |
| `INBOUND_ROUTE_MISSING` | 모든 방문 장소에 진입 경로가 있음 |
| `FINAL_ROUTE_MISSING` | 모든 일차에 해당 날짜 도착지까지 최종 경로가 있음 |
| `DAY_TIME_OVERRUN` | 체류시간과 전체 이동시간이 일차 가용시간 이내 |

Hard Gate 위반 일정은 품질 점수와 관계없이 실패다.

## 실제 Provider 2박 3일 기준선

2026-07-12 ODsay·TMAP 실제 실행 결과는 Hard Gate `PASS`, 품질 `65/100`, 생성시간 `11.77초`다. 경로 해석 50회 중 캐시 적중 24회, 외부 Provider 호출 24회, fallback 0건이었다. 시간·경로 완결성은 통과했지만 부모님·저도보 조건의 부담 장소 2개와 복합 환승 부담 4건 때문에 Iteration 002 목표 80점에 미달했다.

상세 결과: `docs/scenario-results/planner-evaluation-report-2026-07-12.md`

MultiDay 최적화 후 같은 시나리오는 일차별 방문 밀도 `2·2·2`를 유지하면서 Hard Gate `PASS`, 품질 `93/100`, 생성시간 `5.20초`를 기록했다. 부담 장소는 2개에서 0개, 환승은 4회에서 1회로 감소했다. 실제 외부 HTTP는 ODsay 검색 15회, loadLane 15회, TMAP 33회로 총 63회이며 실패 4회, geometry fallback 2개가 확인됐다.

### 기준선 해석 시 주의사항

- 현재 `MOBILITY_FIT`은 대중교통 경로에 포함된 도보 구간이 아니라 전체 구간이 도보인 이동만 감점한다. 개선 후 운영 지표의 전체 도보시간은 73분이지만 점수에는 도보 전용 이동 11분만 반영됐으므로 `93/100`을 릴리스 판정 점수로 사용하지 않는다.
- 여행 기간은 Preview에서 최대 4일이며, 식사·카페·고정 행사를 포함한 일차 전체 방문지는 최대 5곳이다. `PACE_RELAXED`는 가용시간 6시간 이상에서 3곳을, `PACE_PACKED`는 8시간 이상에서 5곳을 목표로 한다. 3시간 미만은 1곳을 목표로 하고, 후보가 부족하면 날짜별 최소 1곳만 보존해 목표를 줄인다. `MultiDayPlanOptimizer`는 최대 20개로 제한한 후보 풀에서 일차별 장소 조합을 생성한다. 전체 목표가 12곳 이하면 비트마스크 동적계획법, 그보다 크면 일차별 상위 조합과 후보 다양성을 제한 폭으로 유지하는 beam DP를 사용한다.
- 실제 Provider 결과는 단일 실행값이다. p50·p95는 같은 시나리오를 반복 실행한 뒤 별도로 기록한다.
- 실제 경로에서 시간 초과가 확인되면 필수 방문지·고정 행사를 보존한 채 선택 활동, 선택 식사 순으로 방문지를 줄여 Hard Gate를 만족시킨다.

## Iteration 003 후보 공동 최적화

- 자동 추천은 활동 목표와 활성 식사 슬롯을 합친 수보다 최대 6곳 큰 후보 풀을 만든다.
- 필수 장소는 후보 제한과 저이동 필터에서도 제거하지 않는다.
- 목적함수는 일차별 추정 이동거리, 식사 시간창 누락, 질문·프롬프트 선호, 이동 부담을 함께 계산한다.
- 식사 장소가 활성 식사 슬롯보다 많을 때도 초과 비용을 부여해 특정 일차 편중을 방지한다.
- 장소 선택과 날짜 배치를 같은 동적계획법에서 비교하고, 선택된 일차 조합의 방문 순서는 `DayRouteOptimizer`가 `PlannerRouteEstimator`의 좌표 기반 비용으로 비교한다.
- `scripts/run-schedule-scenario-check.sh`의 질문, Hard Gate, 경로 순서, 대표 시나리오 검증을 통과한다.
- 다일 날짜 배치안은 Top-K를 보존해 일차 출발·도착 경계의 실제 ODsay 접근성으로 재평가한다. 호출 예산 때문에 모든 조합의 일차 내부 경로까지 평가하지는 않으므로 수학적 전역 최적해를 보장하지 않는다.

## Iteration 006 - 테마 포화와 체험 다양성

- `PlaceExperienceClassifier`가 장소명·카테고리·콘텐츠 타입을 체험 유형과 의미 그룹으로 분류한다. 스키마 변경 없이 계산하며 분류 결과는 저장하지 않는다.
- 후보 풀은 강한 테마 일치 장소만으로 채우지 않고 서로 다른 체험 유형의 대안을 보존한다.
- 테마 일치 보상은 일차별로 첫 번째 100%, 두 번째 50%, 세 번째 15%, 이후 0%로 포화한다.
- 같은 체험과 의미 그룹의 일차 내 반복, 이전 날짜와의 반복, 방문 순서상 연속 반복에 각각 비용을 부여한다.
- 필수 장소와 식사 시간창은 다양성 점수보다 우선한다. 음식점·카페는 일반 반복 감점에서 제외한다.
- 규칙 기반 분류이므로 모호한 장소명은 `OTHER`가 될 수 있다. 향후 적재 시 구조화 태그를 확보하면 분류 입력으로 교체한다.
- 외부 Provider를 호출하지 않는 플래너 단위 테스트와 전체 테스트를 통과했다. 실제 사용자 시나리오의 다양성 점수와 생성시간은 다음 Provider 예산 확보 후 별도로 측정한다.

## Iteration 007 - 다축 경험 프로필과 동선 재진입 감점

- 장소를 단일 대표 테마가 아니라 장소 정체성, 환경, 가능한 경험, 콘텐츠, 분위기의 다축 프로필로 분류한다.
- 해변 축제처럼 행사와 장소 의미가 겹치는 후보는 행사·바다·야경 경험을 함께 반영하고, 해변 산책처럼 배정되지 않은 활동은 별도로 충족 처리하지 않는다.
- 일차 순열 비용에는 첫 권역 전환 비용, 이미 떠난 권역 재진입의 강한 비용, 1km 이상 구간의 방향 전환 비용을 더한다.
- 출발·도착지가 1km 이내인 마지막 복귀는 순환형 일정으로 보고 재진입 예외를 적용한다.
- 순열별 좌표 이동거리와 최단 순열의 비율을 `detourRatio`로 계산한다. 실제 ODsay 경로 기준 튜닝 전에는 비용에 반영하지 않는다.
- 고정 행사 순서 비교도 같은 동선 비용을 적용한다. 행사 시간 도달 가능성이 우선이며, 가능한 행사 순서 중 덜 되돌아가는 경로를 선택한다.

## Iteration 008 - 여행 속도별 밀도와 권장 시간대

- 일차 목표는 활동 장소와 식사 장소를 따로 더하지 않고 전체 방문 항목 기준으로 계산한다. 따라서 식사 때문에 일차 최대 5곳을 초과하지 않는다.
- `PACE_RELAXED`는 6시간 이상일 때 3곳, `PACE_PACKED`는 8시간 이상일 때 5곳을 목표로 한다. 이는 Hard Gate가 아니며 실제 경로와 고정 행사 때문에 감소할 수 있다.
- 3~4곳 목표에는 식사 최대 1곳, 5곳 목표에는 점심·저녁 최대 2곳을 배정해 핵심 관광 경험이 식사 항목으로 대체되지 않게 한다.
- 장소 다축 프로필로 야경·시장·박물관·야외 장소의 권장 방문 시간대를 평가한다. 운영시간 원문을 구조적으로 해석할 수 없으므로 현재는 Soft Penalty만 적용한다.

| `ScheduleHardGateEvaluator` | 필수 장소, 빈 날짜, 시간, 경로 완결성 판정 |
| `ScheduleScoreEvaluator` | Hard Gate 통과 일정의 100점 품질 평가 |
| `SchedulePersistenceService` | 외부 경로 계산 완료 후 짧은 저장 트랜잭션 |

응답·지도 변환은 아직 `ScheduleService`에 남아 있으며 다음 분리 대상이다. 후보 조회와 선호 점수 계산은 `PlaceCandidateProvider`, `PlacePreferenceScorer`로 이동했다.
