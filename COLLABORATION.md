# 협업 규칙

## 브랜치 전략

- `main`: 현재 통합 및 배포 기준 브랜치
- `feature/{기능명}`: 기능 개발
- `fix/{문제명}`: 버그 수정
- `docs/{문서명}`: 문서 전용 변경

작업 브랜치는 최신 `main`에서 생성하고 한 브랜치에는 하나의 작업 목적만 둔다. 추후 팀이 별도 통합 브랜치를 도입하면 이 문서를 먼저 수정한다.

## 작업 전

1. 작업 목적과 완료 조건을 한 문장으로 정리한다.
2. `AGENTS.md` 또는 `CLAUDE.md`의 필수 문서를 읽는다.
3. 관련 API, ERD, 코드, 테스트를 확인한다.
4. API·DB·외부 Provider 영향 여부를 먼저 판단한다.
5. 기존 작업자의 변경을 임의로 되돌리지 않는다.

## 작업 중

- Controller는 HTTP 처리, Service는 비즈니스 로직, Repository는 저장소 접근을 담당한다.
- 외부 API 응답 DTO와 내부 Entity를 분리한다.
- 외부 API 호출에는 타임아웃과 프로젝트 표준 오류 변환을 적용한다.
- API 요청과 응답은 `docs/API_SPEC.md`를 그대로 따른다.
- Entity와 테이블은 `docs/ERD.md`를 그대로 따른다.
- 계약 변경이 필요하면 코드보다 문서 변경 내용을 먼저 명확히 한다.

## 작업 후

1. 변경된 동작에 대한 테스트를 실행한다.
2. API가 바뀌면 `docs/API_SPEC.md`, `docs/API_FIELD_GUIDE.md`, `docs/api-change-log.md`를 수정한다.
3. 엔티티나 DB가 바뀌면 `docs/ERD.md`를 수정한다. migration 도구 도입 후에는 새 migration도 추가한다.
4. 의도하지 않은 파일과 비밀값이 포함되지 않았는지 확인한다.
5. PR에 변경 이유, 테스트 결과, API·ERD 영향, 남은 문제를 적는다.

## 절대 하지 말아야 할 것

- API Key와 인증 정보를 커밋하지 않는다.
- API 또는 ERD 변경을 문서에 반영하지 않은 채 완료하지 않는다.
- migration 도구 도입 후 공유된 migration 파일을 수정하지 않는다.
- Entity를 Controller 응답으로 직접 노출하지 않는다.
- 외부 API의 원본 응답 구조를 도메인 전체로 전파하지 않는다.
- 사용자 요청 없이 CI, 배포, 인프라 설정을 변경하지 않는다.
- 일정 생성 결과에서 `mustVisitPlaceIds`로 지정된 장소를 임의로 제외하지 않는다.

## 1차 스프린트 고정 결정

- 인증과 사용자 도메인은 사용하지 않는다.
- 일정 조회는 파라미터 없이 전체 일정을 반환한다.
- 일정 수정에는 편집 토큰과 `version`을 사용하지 않는다.
- 동시 수정은 마지막 저장 결과가 최종 상태가 된다.
- 장소 검색과 교체는 내부 `places` 데이터를 사용한다.
- 출발지와 도착지 검색은 Kakao Local API를 사용한다.
- 주변 편의시설은 Kakao Local API로 실시간 조회하며 DB에 저장하지 않는다.
- 두루누비 길·코스는 내부 DB에 적재한 뒤 조회하고, 일정 생성의 참고 조건으로만 사용한다.
- PDF와 이미지 저장은 프론트엔드에서 처리한다.
- 예산 계산, 날씨, 공중화장실, 오디오 가이드는 1차 스프린트에서 제외한다.
